The Gaming Room was the client for this project. They wanted a software application for their game, Draw It or Lose It. The main goal was to design a system that could support multiple teams and players, while keeping the game organized and able to run across different platforms.

One thing I did well in this documentation was breaking down the client’s needs into clear design requirements. I explained the system architecture, operating platform options, and how the application should handle users, teams, and games. This helped make the project easier to understand before actually developing the code.

Working through the design document was helpful because it gave me a plan before coding. Instead of just starting with code right away, I had to think about the structure of the program, what the client needed, and what choices would make the software easier to maintain later.

If I could revise one part of my work, I would improve the evaluation section. I would add more detail comparing different operating platforms and explain more clearly why certain choices are better for the client. This would make the recommendation stronger and easier for the client to understand.

I interpreted the user’s needs by focusing on what The Gaming Room wanted the software to do, especially supporting multiple users and keeping game information organized. Considering user needs is important because software should solve the client’s actual problem, not just work from a technical standpoint.

My approach to designing software was to first understand the requirements, then break the system into smaller parts. In the future, I would continue using planning documents, requirements lists, and comparisons of different design options before coding. This process helps create software that is more organized, useful, and easier to update.
